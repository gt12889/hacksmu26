## Issues Found

### Blocker (must fix before pitch)
None — all critical paths functional.

### Warning (nice to fix before pitch)

**1. SNR improvement is negative for all 3 demo cards**

The real-data `metadata.json` shows negative SNR improvement across all noise types:
- Generator: snr_before=20.2, snr_after=20.1 → **-0.1 dB**
- Car: snr_before=20.5, snr_after=19.5 → **-1.0 dB**
- Plane: snr_before=23.4, snr_after=22.3 → **-1.1 dB**

Across the full 212-call batch: 71% show positive improvement, 28% show degradation, average improvement is +0.65 dB. The pipeline uses harmonic dominance (not SNR) as its primary quality metric, and the ML comparison page shows 3-11% harmonic dominance improvement — that framing is more favorable for the pitch. However, a judge looking at the demo card metrics will see SNR *decreasing*, which looks like the denoiser makes things worse.

**Recommendation:** Either (a) swap the displayed metric in demo cards to `harmonic_integrity` or `harmonic_dominance` instead of SNR dB, or (b) add a tooltip/note explaining that SNR is not the right metric for infrasonic bioacoustics (the harmonic comb mask preserves harmonics by design, which may reduce broadband energy slightly).

**2. `types/api.ts` has unused stale field names**

`/frontend/src/types/api.ts` declares `CallResult` with `snr_before_db` and `snr_after_db` — the batch results API does return these exact names, so it is correct. But the file also defines `BatchSummaryResponse` which doesn't appear to be used anywhere in the current frontend. Not a bug, just dead code.

**3. `noise_type` in upload result returns `"mixed"` when source is vehicle**

When uploading the vehicle recording clip, the classifier returns `noise_type: "mixed"` while the batch pre-processed results classify the same file as `"generator"`. This is a classifier sensitivity issue (short 5s clip vs. full recording). Not a bug — the short test clip gives the classifier less signal to work with.

**4. `multi_speaker_demo.png` size is only 218 KB vs 847 KB+ for noise-type demos**

The file exists and serves correctly (200 OK), but it is significantly smaller than the comparable spectrograms. May be lower resolution or fewer panels. Cannot verify visual quality without a browser.
